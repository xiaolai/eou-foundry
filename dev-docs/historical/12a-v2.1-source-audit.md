## Executive verdict

The codebase is a strong concept prototype but not production-ready. The architecture is coherent and the core book-engine idea is well encoded in folders, rules, cards, and scripts. However, the implementation quality is uneven: the seeded book validates, but the validators miss several defects; one Claude hook script has a Python syntax error; one skill has invalid YAML frontmatter; `/new-book` creates a book that fails the project's own validation; and generated manuscript output still contains placeholders and schema-rendering mismatches.

Recommended classification:

- Engine concept: strong
- Repo architecture: promising
- Deterministic code quality: prototype
- Validator reliability: insufficient
- Claude Code integration: partially correct, brittle
- Seeded manuscript quality: scaffold only, not draft-quality prose
- Production readiness: no

## Critical findings

### 1. `.claude/hooks/validate_book_files.py` does not compile

Severity: Critical

Evidence:

```text
*** Error compiling '.claude/hooks/validate_book_files.py'...
  File ".claude/hooks/validate_book_files.py", line 44
    print("
          ^
SyntaxError: unterminated string literal
```

The broken code is:

```python
if problems:
    print("
".join(problems), file=sys.stderr)
    return 1
```

Impact:

- `python3 -m compileall -q scripts .claude/hooks` fails.
- The PostToolUse hook configured in `.claude/settings.json` will fail when invoked.
- Current `validate_workshop.py` does not catch this, so the project can report `OK` while the Claude automation layer is broken.

Fix:

```python
if problems:
    print("\n".join(problems), file=sys.stderr)
    return 1
```

Add hook compilation to `validate_workshop.py`.

### 2. `.claude/skills/device-audit/SKILL.md` has invalid YAML frontmatter

Severity: Critical

The frontmatter line contains an unquoted colon inside the description:

```yaml
description: Audit storytelling devices across a book: metaphor field, analogy bank, mirror artifacts, paired scenes, motifs, and protocol dramatization.
```

YAML parser error:

```text
mapping values are not allowed here
```

Impact:

- The skill may not load in Claude Code.
- The project advertises `/device-audit`, but that skill's metadata is structurally invalid.

Fix:

```yaml
description: "Audit storytelling devices across a book: metaphor field, analogy bank, mirror artifacts, paired scenes, motifs, and protocol dramatization."
```

Add all `.claude/skills/*/SKILL.md` frontmatter parsing to `validate_workshop.py`.

### 3. `/new-book` creates an invalid book according to the project's own validator

Severity: Critical

Command tested:

```bash
python3 scripts/new_book.py \
  --book-id audit-test-book \
  --title 'Audit Test Book' \
  --thesis 'A thesis.' \
  --apparent-proof 'A visible outcome' \
  --hidden-variable 'A hidden variable'

python3 scripts/validate_book.py books/audit-test-book
```

Result:

```text
Validation failed for books/audit-test-book:
- analogy-bank.yml: missing non-empty analogies list
```

Cause:

The template has:

```yaml
analogies: []
```

but `validate_analogy_bank.py` requires a non-empty `analogies` list.

Impact:

- The main expansion workflow is broken.
- Users cannot create a valid new book without manual repair.
- The engine is not truly reusable yet.

Fix options:

1. Relax the validator for early-stage books.
2. Seed one placeholder analogy card and analogy-bank entry.
3. Add book lifecycle stages and validate differently by stage.

Recommended fix: add lifecycle stages.

```yaml
status: architecture | seeded | drafting | production
```

Then validators should enforce stricter requirements only from `seeded` onward.

### 4. The validators pass while generated output contains `TBD`

Severity: High

The seeded book passes:

```bash
python3 scripts/validate_book.py books/the-perfect-assignment
# OK
```

But generated output contains unresolved placeholders:

```text
books/the-perfect-assignment/manuscript/chapters/ch-018-fluency-weak-signal.md: - TBD
books/the-perfect-assignment/manuscript/chapters/ch-022-ai-as-critic.md: Scene A: executor / before first attempt → TBD
books/the-perfect-assignment/manuscript/chapters/ch-022-ai-as-critic.md: Scene B: critic / after first attempt → TBD
```

Impact:

- Validation gives false confidence.
- Exports may include obvious scaffold artifacts.
- The engine's own thesis is weakened: it produces polished-looking output without enough process-quality assurance.

Fix:

- Add `check_no_tbd_in_manuscript.py`.
- Add the check to `validate_book.py` or a stricter `validate_book --stage production` mode.

### 5. Schema mismatch: paired-scene renderer expects `learner_result`, but card uses `result`

Severity: High

Renderer in `scripts/compile_chapter.py`:

```python
f"Scene A: {a.get('ai_role', 'TBD')} / {a.get('timing', 'TBD')} → {a.get('learner_result', 'TBD')}"
```

Actual card:

```yaml
scene_a:
  ai_role: executor
  timing: before first attempt
  result: "polished artifact, weak residue"
```

Impact:

- Output contains `TBD` despite a complete card.
- `validate_story_devices.py` does not validate paired-scene internal schema.

Fix:

Either change cards to:

```yaml
learner_result: "polished artifact, weak residue"
```

or change renderer to accept `result`.

Better: define a single JSON Schema / Pydantic model and use it in renderer and validator.

## High-priority design findings

### 7. Validation is too shallow and partly string-based

Current validation checks field presence more than semantic correctness. The hook validator even checks raw substrings such as `"id:"` rather than parsing YAML.

Risk:

- False positives: invalid content passes.
- False negatives: comments or nested text can satisfy substring checks.
- Different scripts may define different expectations for the same object.

Recommendation:

Create real schema models:

```text
schemas/
├── chapter-card.schema.json
├── metaphor-card.schema.json
├── analogy-card.schema.json
├── mirror-artifact-card.schema.json
├── paired-scene-card.schema.json
├── motif-card.schema.json
└── protocol-scene-card.schema.json
```

or use Pydantic models in Python.

---

### 8. Hook commands are brittle with respect to current working directory

`.claude/settings.json` uses commands like:

```json
"command": "bash -lc 'python3 .claude/hooks/validate_book_files.py'"
```

This is safe only if the hook runs from the workshop root. The README recommends starting Claude Code from the workshop root, but the conceptual architecture also supports working from `books/<book-id>/`.

Observed locally:

```bash
cd books/the-perfect-assignment
python3 ../../.claude/hooks/validate_metaphor_files.py
# exits 0 while checking nothing, because it expects a root-level books/ directory
```

Recommendation:

- Make all hooks locate the workshop root by walking upward.
- Use `scripts/_common.py` root-detection logic in hooks too.
- Configure commands through a wrapper:

```bash
python3 scripts/run_hook.py validate_book_files
```

---

### 9. There is no test suite

The project has validators, but no tests for the validators, compilers, hook scripts, or new-book workflow.

Minimum useful tests:

```text
tests/test_new_book_validates.py
tests/test_compile_chapter_no_tbd.py
tests/test_skill_frontmatter_valid.py
tests/test_hooks_compile.py
tests/test_story_device_schema.py
tests/test_seeded_book_validates.py
```

This would have caught the top four defects.

### 10. Generated manuscript is formulaic and should be classified as scaffold, not draft prose

The seeded manuscript contains repeated generic language:

```text
A concrete moment around the assignment makes the old reading feel reasonable...
Something about the artifact, the learner, or the process does not fit...
See book/fair-clues.yml for the clue assigned to this chapter.
The chapter reveals one part of learning residue...
```

Counts in manuscript chapters:

```text
A concrete moment around: 36
Something about the artifact: 36
See book/fair-clues.yml: 72
The chapter reveals one part: 36
The apparent proof does not disappear: 36
```

Impact:

- The compiler currently produces a mechanical placeholder manuscript.
- This is acceptable for a scaffold, but not for a book draft.

Recommendation:

Rename current compiler mode to:

```text
compile_chapter_scaffold.py
```

Then create a separate prose-drafting skill that consumes the scaffold and writes real prose under strict audit.

## Architecture audit

Strengths:

- The workshop-monorepo model is correct.
- Separation between engine, book-specific source, cards, manuscript, audits, and exports is sound.
- A2B Engine V2.1 is clearly represented in `engine/`, `dev-docs/`, `.claude/rules/`, and book metadata.
- The project correctly treats each book as a workbench under `books/<book-id>/`.
- The card-based workflow prevents purely prompt-driven manuscript generation.

Weaknesses:

- The engine is over-documented relative to the amount of tested executable code.
- There is no canonical schema layer, so rules, validators, compilers, and card files drift.
- The seeded book is over-seeded with 38 chapter cards but many are generic; breadth arrived before depth.
- `exports/` contains generated Markdown, but `.gitignore` only ignores root `exports/*.docx` and `exports/*.pdf`, not `books/*/exports/`.

Recommended structural addition:

```text
schemas/
tests/
Makefile
scripts/run_all_checks.py
scripts/run_hook.py
```

## Claude Code integration audit

Strengths:

- Project skills are used appropriately for reusable procedures.
- Root-level skills are the right fit for a workshop monorepo.
- Commands are kept as thin aliases, which is reasonable.
- Subagents are conceptually useful and well named.

Problems:

- One skill frontmatter is invalid YAML.
- Hook scripts are not tested by workshop validation.
- Hook path assumptions are brittle.
- Several skills instruct Claude to write audits, but there is no deterministic check that audits exist before export.
- Skill descriptions and rule files may become context-heavy; there is no budget control or skill override policy.

Recommendation:

Add a dedicated Claude config validator:

```bash
python3 scripts/validate_claude_config.py
```

It should check:

- all skill frontmatter parses
- all command frontmatter parses
- all agent frontmatter parses
- all hook scripts compile
- all hook paths exist
- settings JSON parses
- hooks can locate root from both root CWD and book CWD

## Security audit

Risk level: moderate-low for current prototype.

Strengths:

- `.claude/settings.json` denies reads of `.env`, `.env.*`, `secrets/**`, and `private/**`.
- The scripts do not make network calls.
- Generated content is local.

Concerns:

- Many skills allow `Bash`, which increases blast radius if a future skill is poorly written.
- Hooks run automatically after write/edit operations.
- Hook commands are `bash -lc`, which is common but increases shell-injection sensitivity if interpolated arguments are later added.
- `new_book.py` copies templates and replaces text without checking for path traversal-like user inputs, though `slug()` reduces risk for `book_id`.

Recommendations:

- Use Python wrapper scripts instead of shell fragments where possible.
- Keep `Bash` out of most skills unless necessary.
- Add `scripts/security_audit.py` to list all skills with Bash access.
- Add `.gitignore` entries for `books/*/CLAUDE.local.md` and `books/*/.claude/settings.local.json` if book-local configs are later introduced.

## Maintainability audit

Strengths:

- Directory layout is comprehensible.
- File names mostly express intent.
- Scripts are small and readable.
- `_common.py` centralizes YAML loading and book resolution.

Problems:

- There are duplicate/near-duplicate scripts: `check_story_devices.py` and `check_storytelling_devices.py`.
- Validation logic is distributed and inconsistent.
- There are no dataclasses or typed models for core card types.
- `compile_chapter.py` hardcodes story-device rendering logic in a single function; it will become difficult to extend.

Recommendation:

Refactor toward:

```text
book_workshop/
├── models.py
├── schemas.py
├── validators.py
├── renderers/
│   ├── chapter.py
│   ├── metaphor.py
│   ├── analogy.py
│   ├── mirror_artifact.py
│   └── paired_scene.py
└── cli.py
```

Keep the current scripts as thin CLI wrappers.

## Content-engine audit

Strengths:

- The governing reversal is clear.
- Metaphor/analogy rules are strong and disciplined.
- The idea of preserving process evidence is consistently represented.
- The seeded book has a coherent macro-architecture.

Problems:

- Chapter cards are too templated.
- Many chapters do not yet contain real scenes.
- Evidence files are placeholders rather than populated research bases.
- Claim cards are not fully integrated into chapter compilation.
- The compiler produces explanatory scaffold sections instead of narrative prose.

Recommendation:

Separate four artifact levels:

```text
1. Architecture card
2. Chapter scaffold
3. Prose draft
4. Audited manuscript chapter
```

Current output is level 2. Do not call it level 3.

## Bottom line

This is a valuable engine prototype, not a reliable production workshop yet. The main design is worth preserving. The next engineering phase should not add features. It should harden the system: canonical schemas, tests, hook validation, root-aware scripts, no-placeholder export gates, and lifecycle-aware book validation.
